Goal

A single web page that automatically places a child into the correct Making Math Meaningful level. It replaces manual use of the paper "Levels Chart" with an adaptive quiz: the child works real, auto-graded problems, and the page recommends the exact level the chart would — the first level whose skills the child cannot yet do.

Hosting is undecided; the deliverable is a self-contained static page that works anywhere (no backend, no build step). Style must fit the existing cornerstonecurriculum.com look — cream, deep maroon, classic serif.

User flow — four screens
  1. Welcome (parent). What the quiz does ("find the perfect starting level, about 5–10 minutes"), the program's philosophy (levels are not grades; children auto-advance when ready), a note to have pencil and scratch paper ready, and one required input: child's age ("4 or younger" through "14 or older"). Age only seeds the starting rung — it never decides the result. "Not sure? Pick your best guess — the quiz adapts."
  2. Hand-off. "Now hand the device to your child." For young children, the parent sits alongside and reads problems aloud. Checkpoint K items are explicitly parent-assisted.
  3. Quiz. One problem per card: large type, progress dot trail, an "I don't know yet" button (silent miss — no forced guessing), encouraging transitions between sets. No red X's, no visible scoring, no timer. Wrong answers are never announced.
  4. Results (parent). Recommendation, skill breakdown, philosophy note, product link, start-over.
Placement engine

Checkpoints

Eight (K–7), each a set of 4–6 auto-graded problems; every skill line of that chart row is represented by at least one problem.

Seeding

Expected checkpoint by age ≈ (age − 5); seed one rung below it so the first set is a likely pass and builds confidence. Formula: seed = clamp(age − 6, K, 6). Mapping: ≤5 → K, 6 → K, 7 → 1, 8 → 2, 9 → 3, 10 → 4, 11 → 5, 12+ → 6. Max seed is 6, so checkpoint 7 is always reached by passing 6.

Pass rule

A checkpoint is passed when the child misses at most one problem in the set (≈80%+). "I don't know yet" counts as a miss.

Adaptive walk

  • First result pass → step up one rung at a time until the first fail (that rung is the recommendation), or checkpoint 7 passes (→ Geometry/Algebra II outcome).
  • First result fail → step down until the first pass (the lowest failed rung above it is the recommendation), or checkpoint K fails (→ Level K).

The walk is monotonic, so it always terminates; it mirrors the chart's own ladder logic. Typical run: 2–3 checkpoints, 8–15 problems.

State

Quiz state persists in sessionStorage so an accidental refresh doesn't lose progress. "Start over" clears it.

Problem bank

One fixed problem per slot (variant pools are a future enhancement). Written in the curriculum's conceptual style — word problems and real contexts wherever the chart says "word problems."

Input types: numeric entry for computation (digits only, no empty submits); multiple choice for notation-heavy items (fractions, symbols, factoring). Each problem records the skill line it tests.

Results page
  • Recommendation front and center, in the chart's own words: "Begin using Level 4."
  • Skill breakdown, positively framed:
    • ✓ "Your child demonstrated" — skill lines from passed checkpoints, plus skill lines in the failed checkpoint whose problems were all answered correctly.
    • ○ "Level N will teach" — the remaining skill lines of the recommended level.
  • Philosophy note from the chart: auto-advance when ready; age recommendations adapt to your family.
  • Product link — one clear button to the recommended level's page on cornerstonecurriculum.com.
  • Start over for a second child.
Look & feel
  • Palette: cream background, deep maroon headings/actions, warm gray text — as CSS custom properties. Color used semantically; no green/red correctness cues mid-quiz.
  • Typography: two families max — classic display serif for headings (EB Garamond or similar), highly readable face for problems/UI, fluid clamp() sizes, font-display: swap.
  • Layout: single centered column; large problem cards; subtle progress dots; parent screens slightly bookish, child cards cleaner and bigger.
  • Motion: gentle fades/transforms only (compositor-friendly), disabled under prefers-reduced-motion.
  • Responsive & accessible: 320 px and up, touch-friendly, semantic HTML, labeled inputs, focus management, keyboard operable, WCAG AA contrast.
Architecture

Static site — no framework, no build step, no backend:

/                        (repo root = deployable root)
├── index.html           # all four screens, semantic markup
├── css/styles.css       # tokens + styles
├── js/data.js           # chart data: checkpoints, skills, problems, links
├── js/engine.js         # pure placement logic — no DOM
├── js/app.js            # screen flow, rendering, input, sessionStorage
└── tests/engine.test.js # node:test suite

Separating data.js lets the curriculum author tweak problems and wording without touching logic. engine.js is pure functions — directly unit-testable.

Error handling & edge cases
  • Invalid/empty numeric input: submit disabled until valid; no error flashes at the child.
  • Accidental refresh: state restored from sessionStorage.
  • "I don't know yet": silent miss; prevents forced guessing from inflating placement.
  • Fail at K / pass at 7: explicit terminal outcomes.
  • Commas in large numbers stripped before grading.
  • No quiz-time network dependencies beyond fonts (system-font fallbacks provided).
Testing
  • Engine (node:test): seeding for every age; pass-rule boundaries; walk scenarios producing each of the 9 outcomes; pass-all/fail-all from every seed; state round-trip.
  • Data validation: every skill line has ≥1 problem; every problem has a valid answer key and skill reference; every level has a product link.
  • Manual/visual: 320/375/tablet/desktop; reduced motion; keyboard-only run-through.
Out of scope (v1)

Email capture / mailing list, printing/PDF export, analytics, problem variant pools, multiple-child profiles, server-side anything.
